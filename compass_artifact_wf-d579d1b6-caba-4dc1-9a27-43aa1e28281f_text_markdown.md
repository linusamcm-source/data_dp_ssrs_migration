# Migrating PBIRS/SSRS on Azure VMs: Symmetric Key & Stored-Credential Strategy — ADF vs. PowerShell/Python

## TL;DR
- **The symmetric-key-aware native/scripted approach is the only correct method for the encrypted ReportServer catalog; Azure Data Factory cannot migrate the encrypted system database.** The canonical Microsoft path is: back up the encryption key → back up `ReportServer`/`ReportServerTempDB` → restore both databases (same names) on the target → restore the encryption key on the target so stored credentials and connection strings stay decryptable.
- **ADF has a genuine but narrow role:** moving the *underlying report data sources* (data warehouses, marts, line-of-business tables the reports query), not the encrypted SSRS catalog. ADF's SQL connector copies row-level table data; it cannot reproduce the application-layer-encrypted blobs in the `Keys`/`DataSource` tables, which are only meaningful with the matching symmetric key.
- **For SQL Server on Azure VMs (IaaS), use native SQL `BACKUP/RESTORE TO URL` via Azure Blob Storage** to move `.bak` files between VMs, and automate the whole runbook with the `ReportingServicesTools` PowerShell module (Backup/Restore-RsEncryptionKey) plus Python for orchestration, REST API re-keying, and validation.

## Key Findings

### 1. Why the encryption key is the crux
SSRS/PBIRS stores sensitive data — stored data-source credentials, connection strings, subscription delivery credentials — encrypted inside the `ReportServer` database. Per Microsoft Learn ("Configure and Manage Encryption Keys"), "encryption is supported through a combination of public, private, and symmetric keys that are used to protect sensitive data." The symmetric key encrypts the data; per the "Store Encrypted Report Server Data" page, "This symmetric key is itself encrypted using the public key of an asymmetric key pair generated by Windows. The private key is held by the Report Server Windows service account." Public/private keys are "created by the operating system" and SSRS's own tools "are used to manage the symmetric key only."

The practical consequence: if you simply restore the `ReportServer` database onto a new VM, the new host's Report Server service has a *different* OS-generated key pair and cannot decrypt the symmetric key — so all encrypted content (stored credentials/connection strings) is inaccessible. The "Initialize a Report Server" page makes the binding explicit: "A report server will be initialized only if there is a match between the installation identifier and the public key… If the match fails, the report server is disabled, in which case you may be required to apply a backup key or delete the encrypted data."

You must therefore restore the backed-up symmetric key. On restore, per Microsoft Learn: "The symmetric key is retrieved from the password-protected backup file. The symmetric key is encrypted by using the Report Server Windows service public key. The newly encrypted symmetric key is stored in the report server database. The previous symmetric key data is deleted." This re-binds the existing encrypted content to the new host — which is why the key backup `.snk` file plus its password are mission-critical.

### 2. Canonical Microsoft migration process (native mode → PBIRS)
Microsoft's "Migrate a report server installation" (Power BI docs) and "Migrate a Reporting Services Installation (Native Mode)" (SQL docs) define the same ordered process:
1. **Back up the encryption key** (Report Server Configuration Manager → Encryption Keys → Backup, producing a password-protected `.snk`; or `rskeymgmt -e`; or `Backup-RsEncryptionKey`).
2. **Back up `ReportServer` and `ReportServerTempDB`** using any supported SQL backup method (full recovery model recommended for `ReportServer`).
3. **Install the target** PBIRS instance (instance name is always `PBIRS`; there is no in-place upgrade from SSRS to PBIRS — migration is mandatory). You cannot install PBIRS on the same server you are migrating *from* when going PBIRS→PBIRS.
4. **Restore both databases** to the target SQL instance, keeping the same database names (`ReportServer`).
5. **Configure the target** via Report Server Configuration Manager to point at the restored database.
6. **Restore the encryption key** on the target (`rskeymgmt -a -f file.snk -p <password>`, or Configuration Manager → Encryption Keys → Restore, or `Restore-RsEncryptionKey`).
7. **Clean up the scale-out list:** after restore, the `Keys` table may list both old and new instances. On Standard edition this triggers a "scale-out deployment is not supported" error; delete the stale source-server row (Configuration Manager, or delete the row in the `Keys` table for scale-out scenarios).

Critical detail: SQL Agent jobs backing subscriptions live in `msdb`, not in the SSRS databases. They are *not* moved by the database restore. The report server recreates subscription jobs automatically on startup; Microsoft explicitly advises *not* to move the old SQL Agent jobs.

### 3. The lost-key scenario (delete and re-create)
If the key backup or its password is lost, you cannot decrypt existing content. The only recovery is to delete the encrypted content:
- `rskeymgmt -d` deletes all symmetric key information and all encrypted content from the database (unencrypted content — reports, folders, RDLs — is left intact); the server re-initializes with a fresh symmetric key.
- You then must manually re-enter every secret: for each shared data source retype the connection string and stored credentials; for each report/data source with stored credentials retype username/password; for each data-driven subscription retype the subscription-database credentials; for subscriptions using encrypted delivery (file share, third-party) retype those credentials.

**When is delete-and-recreate appropriate vs. migrating the key?** Migrating the key is *always* preferred when you have it — it is faster, lossless, and avoids re-entering potentially hundreds of credentials. Delete-and-recreate is appropriate only when: (a) the key/password is genuinely lost; (b) you are intentionally rotating credentials anyway (e.g., security incident, service-account change); or (c) the credential inventory is small enough that scripted re-entry is trivial. Note `rskeymgmt -s` resets/re-creates the symmetric key on a *working* server (re-encrypts all content) — different from `-d`, which destroys encrypted content.

### 4. SQL Server on Azure VM (IaaS) specifics
- **Backup transport:** The cleanest way to move `.bak` files between Azure VMs is native **SQL Server Backup to URL** targeting Azure Blob Storage. Use `BACKUP DATABASE [ReportServer] TO URL = 'https://<acct>.blob.core.windows.net/<container>/ReportServer.bak' WITH COMPRESSION, CHECKSUM` and a matching `RESTORE ... FROM URL` on the target VM. Authentication is via a SQL Server `CREDENTIAL` holding either a Shared Access Signature or the storage account key. Per Microsoft Learn ("SQL Server Backup to URL for Azure Blob Storage"): "For SQL Server 2016 and later, block blob is preferred. If the storage key is used in the credential, page blob is used; if the Shared Access Signature is used, block blob is used." Starting **SQL Server 2022 Cumulative Update 17 (CU17)**, per Microsoft Learn ("Backup and Restore to URL Using Managed Identities"), "you can use managed identities with SQL Server credentials to back up to and restore SQL Server on Azure VM databases from Azure Blob storage" — requiring "A SQL Server on Azure VM with SQL Server 2022 CU17 or later registered with the SQL IaaS Agent extension," Microsoft Entra authentication, and the primary managed identity assigned the `Storage Blob Data Contributor` role on the storage account.
- **Azure Backup vs native:** Azure Backup / the SQL IaaS Agent's Automated Backup protects the whole DB for DR but is less convenient for a controlled, point-in-time cross-server migration than native `BACKUP/RESTORE TO URL`, which gives you a portable `.bak`. Note the SQL IaaS Automated Backup only covers the default instance (`MSSQLSERVER`), not named instances.
- **Network/firewall:** Use a storage account in the **same Azure region** as the VMs to avoid egress charges and latency. Open outbound TCP 443 to the blob endpoint; watch for proxy servers killing multi-threaded backup connections, and TLS version mismatches on the storage account. If copying report-data directly VM-to-VM via ADF, the self-hosted IR makes only outbound 443 connections to Azure, but SQL connectivity (TCP 1433) and Windows Firewall rules between IR and the SQL VM still apply.
- **Encryption key file:** The `.snk` must travel separately and securely (not in the same container as the `.bak`, per Microsoft guidance). Store it (and its password) in Azure Key Vault.

### 5. Can ADF do this? Honest assessment
**No — not for the encrypted catalog or the symmetric key.** Reasons, grounded in how ADF works:
- ADF's SQL Server connector performs **row-level copy activity** (SELECT from source tables → INSERT into sink tables). It is designed for data movement/ETL, not for cloning an application's encrypted system database with fidelity.
- The symmetric key is an **OS/service-level construct** managed via WMI/RPC on the report server host. The PowerShell `Restore-RsEncryptionKey` cmdlet invokes the WMI `MSReportServer_ConfigurationSetting.RestoreEncryptionKey` method; `rskeymgmt` connects to the local Report Server Windows service over RPC and, per Microsoft Learn, "can't connect to remote instances of the Report Server Windows service so it can't be used to manage the encryption keys of a remote report server instance." ADF has no mechanism to extract, re-encrypt, or apply this key.
- Even if ADF copied every row of the `ReportServer` `Keys`, `DataSource`, and `Catalog` tables verbatim, the encrypted blobs would be **useless on the target** because they are bound to the source's OS key pair. ADF cannot re-key them. (This is analogous to ADF's documented Always Encrypted handling, where the column master key must live in Key Vault and be referenced — ADF does not itself re-encrypt application secrets.)
- ADF/Copy-Database tooling also categorically excludes system databases: the Copy Database Wizard "cannot be used to copy or move databases that are system databases," and ADF copy is table-level, not database-level.

**What ADF *is* genuinely good for here:**
- Migrating/replicating the **underlying data sources** the reports consume — e.g., lifting a data warehouse or operational tables from the source SQL VM to a new target (data warehouse modernization, region move, or consolidation) using the SQL Server connector + self-hosted integration runtime.
- Ongoing **ETL/data refresh** that feeds the reporting datasets after migration.
- It is *not* the tool to migrate the SSRS catalog, the encryption key, subscriptions, or stored credentials.

### 6. The robust PowerShell + Python scripted approach
**PowerShell (primary engine):**
- **`ReportingServicesTools`** (Microsoft, open-source on GitHub; install via `Install-Module ReportingServicesTools` or `Invoke-Expression (Invoke-WebRequest https://aka.ms/rstools)`). Key cmdlets:
  - `Backup-RsEncryptionKey -Password '<pwd>' -KeyPath 'C:\rs\key.snk'` and `Restore-RsEncryptionKey -Password '<pwd>' -KeyPath 'C:\rs\key.snk'` (with `-ReportServerInstance`, `-ReportServerVersion`, `-ComputerName` for non-default instances). These wrap the WMI `MSReportServer_ConfigurationSetting` Backup/RestoreEncryptionKey methods.
  - Content-level migration (alternative/supplement to DB restore): `Out-RsCatalogItem`/`Out-RsFolderContent` (download) and `Write-RsCatalogItem`/`Write-RsFolderContent` (upload), or the REST equivalents `Out-RsRestCatalogItem`/`Write-RsRestCatalogItem`.
  - Data-source re-keying: `Get-RsRestItemDataSource` / `Set-RsRestItemDataSource` (REST v2.0, handles paginated reports and Power BI reports / `DataModelDataSource`), plus `Set-RsDataSource` / `Set-RsDataSourcePassword` (SOAP).
- **Known gotchas:** `Restore-RsEncryptionKey` historically mis-derived the service name on SSRS 2017 (instance `SSRS` → service `SQLServerReportingServices`) and fails to restart the service when run remotely with alternate credentials (GitHub issues #159, #228) — run it locally on the report server host. Also confirm the WMI-encoded instance name (extra `_5f` characters can appear) before scripting.
- **`rskeymgmt.exe`** (located in `...\Microsoft SQL Server\nnn\Tools\Binn`): `-e` extract/backup, `-a` apply/restore, `-d` delete encrypted content, `-s` reset key, `-r` remove an instance from scale-out, `-j` join scale-out. Must run locally; works only against the local Report Server service.
- **`rs.exe`** with the Microsoft `ssrs_migration.rss` script (in `microsoft/sql-server-samples`): copies catalog content (reports, datasets, data sources, folders, roles) server-to-server via the SOAP endpoint — supports SQL 2008 R2+ and PBIRS, same-mode or cross-mode. **Passwords are NOT migrated** by `rs.exe`; stored credentials must be re-entered afterward. This is the content-migration alternative when you deliberately do *not* want to move the database/key (e.g., a clean rebuild).
- **Azure:** `Backup to URL` via T-SQL/`Invoke-Sqlcmd`; `Az.KeyVault` (`Set-AzKeyVaultSecret`/`Get-AzKeyVaultSecret`) for secrets.

**Python (orchestration, re-keying, validation):**
- Call the **SSRS/PBIRS REST API v2.0** (`http(s)://<server>/Reports/api/v2.0/...`) with `requests` + `requests_ntlm`'s `HttpNtlmAuth` (or `requests-negotiate-sspi` for Windows Integrated). Endpoints: `GET /CatalogItems`, `GET/PATCH /DataSources`, `GET/PATCH /Reports({id})/DataSources` and `/PowerBIReports({id})/DataSources`.
- **Re-keying payload nuance:** When setting stored credentials on a Power BI report data source you must set `CredentialRetrieval = "store"` *and* populate both `CredentialsInServer` (UserName/Password/UseAsWindowsCredentials) *and* the `DataModelDataSource` Username/Secret, or scheduled refresh fails. The `SetItemDataSources`/PUT endpoint expects an *array* of DataSource objects and requires `CredentialRetrieval` even though the published schema understates it.
- Python's natural fit: orchestrating the end-to-end runbook, pulling secrets from Azure Key Vault (`azure-keyvault-secrets` + `azure-identity`), driving idempotent credential re-entry from an inventory, and post-migration validation (render-test every report, confirm data-source connectivity, check subscriptions).

### 7. Stored credentials & connection strings — handling and re-keying
SSRS data sources support four credential modes:
- **Stored credentials** ("Credentials stored securely in the report server") — username/password (SQL login *or* Windows account) stored, per Microsoft Learn, "in reversible encryption in the report server database"; required for unattended operations like subscriptions and scheduled refresh. The "Use as Windows credentials" flag distinguishes a Windows account from a database login; "Impersonate/Set execution context" can be layered on.
- **Windows integrated security** — passes the requesting user's token (needs Kerberos delegation for the double-hop).
- **Prompt** (credentials supplied by the user at runtime).
- **No credentials** — relies on the unattended execution account.

Only stored/encrypted values are protected by the symmetric key, so those are exactly what break on a key-less migration. **Inventory** them up front by querying the catalog (`Get-RsRestItemDataSource` across all items, or the `Catalog`/`DataSource` tables) to enumerate every data source, its `CredentialRetrieval` mode, and connection string. **Secure the secrets** during migration in **Azure Key Vault** (`Set-/Get-AzKeyVaultSecret`), never in plaintext scripts; in ADF linked services, reference Key Vault rather than inline secrets. **Re-key** either by restoring the symmetric key (no re-entry needed) or, in the lost-key/rebuild path, by scripting `Set-RsRestItemDataSource` / `Set-RsDataSourcePassword` from the Key Vault-sourced inventory.

### 8. Version compatibility
- **Target ≥ source.** SSRS upgrades the `ReportServer` schema in-place on first startup: each time the service starts it checks the DB schema version and auto-upgrades a lower version to match — this works across SQL/SSRS versions (2016→2019, etc.) and even across database-engine versions hosting the catalog. You cannot point a *newer* database schema at an *older* report server.
- **SSRS→PBIRS is always a migration** (no in-place upgrade). Per Microsoft migration guidance, migrations from **SQL Server 2022 Reporting Services and later require Power BI Report Server (May 2025) or later**; SQL Server 2008 Reporting Services and later are supported as migration sources.
- **Strategic context (June 2026):** Microsoft has consolidated on-premises reporting under PBIRS. Per the "Reporting Services Consolidation FAQ," "SQL Server Reporting Services (SSRS) 2022 will continue to receive security updates and support through January 11, 2033. However, no new versions will be released beyond SSRS 2022" — and no SSRS ships with SQL Server 2025. All new RDL features now go to PBIRS, which ships under Microsoft's Modern Lifecycle Policy with updates three times per year (January, May, September). PBIRS is now licensable with SQL Server 2025 Standard or Enterprise: the FAQ states "Power BI Report Server is available for customers running SQL Server 2025 Enterprise and Standard editions… Customers who purchase a qualifying SQL Server 2025 license can use the same SQL Server 2025 key to install and use PBIRS."
- **Build to target:** As of this writing the **May 2026** PBIRS build exists (Download Center id 105944), though Microsoft Learn's download page still labels **January 2026** as the latest GA version — confirm the current GA build before deploying. Either build requires a **SQL Server 2014 SP3 or later** database engine (and Analysis Services 2014 SP3+) to host the catalog. SQL Server 2025 product-key support was added incrementally: PBIRS build 15.0.1119.121 (Dec 1, 2025) "Added support for SQL Server 2025 Product IDs (keys)," and 15.0.1120.113 (Jan 21, 2026) added support for the SQL Server 2025 Enterprise Core key.
- **Collation gotcha:** Direct restores across very old→new versions can fail on legacy LOB types (`ntext`) vs UTF-8/UTF-16 collation mismatches (documented 2012→2019 failures). The field-proven workaround is a **stepped migration** (e.g., 2012 → 2017 → 2019/PBIRS).

## Details — Recommended end-to-end runbook (key-migration path)

**Phase A — Pre-migration (no downtime; repeatable):**
1. Inventory: enumerate reports, shared/embedded data sources (+ `CredentialRetrieval` mode and connection strings), subscriptions, roles/permissions, and custom config (`RSReportServer.config` web settings, execution account, SMTP). Save the inventory; push every secret into Azure Key Vault.
2. Analyze source load (CPU/RAM, `ExecutionLog3`) to size the target VM.
3. Provision the target Azure VM, OS, networking/NSG, service accounts; install the target PBIRS ("Install report server only" — do not configure yet).
4. Back up the **encryption key** (`Backup-RsEncryptionKey`) → store `.snk` + password in Key Vault.
5. Test-restore a copy of `ReportServer`/`ReportServerTempDB` to the target and dry-run the key restore + report rendering, repeating as needed (all pre-outage).

**Phase B — Migration (outage window):**
6. Quiesce the source (disable `IsWebServiceEnabled` / stop subscriptions) to get a consistent backup.
7. `BACKUP DATABASE [ReportServer]`/`[ReportServerTempDB] ... TO URL` (Blob, `WITH COMPRESSION, CHECKSUM, COPY_ONLY` as appropriate).
8. `RESTORE ... FROM URL` on the target VM, keeping identical DB names; set DB compatibility level as needed.
9. Point the target PBIRS at the restored DB (Configuration Manager → Database).
10. **Restore the encryption key** on the target (`Restore-RsEncryptionKey` / `rskeymgmt -a`), run locally on the target host.
11. Configure Web Service URL, Web Portal URL, SSL cert, SMTP, execution account; delete the stale source instance from the scale-out `Keys` list.

**Phase C — Post-migration validation:**
12. Render-test reports (browser + scripted/Python), confirm each data source connects, export to PDF/Excel.
13. Confirm subscriptions exist and their SQL Agent jobs were auto-recreated in `msdb`; fire a test subscription.
14. Verify folder/role security migrated; re-apply role assignments if any were lost.
15. Verify Power BI (PBIX) reports and any AS data sources; ensure report authors use the matching version of Power BI Desktop for Report Server.
16. Cut over DNS/load balancer to the new server (a DNS CNAME alias avoids changing client URLs); keep the source intact as rollback until sign-off, then decommission.

**Rollback:** Because the source remains untouched until cutover, rollback is simply re-pointing DNS back. Keep the `.snk`, `.bak` files, and Key Vault secrets until UAT sign-off.

## Recommendations
1. **Default to the native/scripted key-migration path** (Phases A–C). Use `ReportingServicesTools` for the key + content, native `BACKUP/RESTORE TO URL` for the databases, Azure Key Vault for the `.snk`/password/credentials, and Python for orchestration + validation. This is the technically correct approach for the encrypted catalog.
2. **Use ADF only for the underlying report data** — migrating/replicating the data warehouses and source tables the reports query, via the SQL connector + self-hosted IR. Do **not** attempt to move the `ReportServer` catalog, the symmetric key, or stored credentials through ADF.
3. **Always migrate the key when you have it.** Reserve `rskeymgmt -d` (delete encrypted content) for genuine lost-key situations or deliberate credential rotation; pair it with scripted `Set-RsRestItemDataSource` re-entry from your Key Vault inventory.
4. **Choose the content-rebuild path (`rs.exe` + re-keying) instead of DB restore** when you want a clean catalog, are crossing a wide version gap that risks collation/schema errors, or are consolidating multiple servers — accepting that you must re-enter all stored credentials.
5. **Decision thresholds:** If credential count is large (dozens+) and the key is available → key restore. If a direct cross-version DB restore throws LOB/collation errors → stepped migration (interim version) or `rs.exe` content copy. If migrating data sources to a new warehouse → add ADF. If the key/password is unrecoverable → `rskeymgmt -d` + scripted re-key.
6. **Target version:** Land on the current GA PBIRS build (verify January vs. May 2026 GA at deploy time) given SSRS's end of the line; ensure SQL 2022 RS or later sources go to PBIRS May 2025+.

## Caveats
- Microsoft does not state in a single sentence that "moving the database to a new machine makes content undecryptable"; this is the documented logical consequence of the OS-bound asymmetric-key protection ("The private key is held by the Report Server Windows service account") plus the installation-ID/public-key match requirement enforced at report server initialization.
- `rskeymgmt` uses an RPC endpoint to the local service; the PowerShell `Restore-RsEncryptionKey` uses the WMI `MSReportServer_ConfigurationSetting.RestoreEncryptionKey` method — both are local-host operations, but they are different transports; don't conflate them.
- SQL Agent subscription jobs (`msdb`) are recreated by the report server, but practitioners report cases where they fail to recreate (often an encryption-key symptom) — validate explicitly.
- Some practitioner blogs and community posts cited are field reports, not official guidance; treat version-specific workarounds (e.g., the 2012→2017→2019 stepping) as experience-based.
- PBIRS licensing/edition behavior under SQL Server 2025 is still being clarified by Microsoft (e.g., Standard-edition feature parity, and SQL 2025 product-key support that arrived only in Dec 2025–Jan 2026 builds) — confirm current entitlement and the latest GA build before purchasing or deploying.
